# Bus Engine Website And Docs Rewrite

## Session Context

The current work is a public presentation rewrite for Bus Engine across the
commercial `busdk.com` site and the public `docs` site. The operator redirected
the Bus Engine copy away from a generic AI engineering assistant story and
toward a specific product hierarchy: BusDK is the Business Development Kit
ecosystem, Bus Engine is the commercial Linux system-engineering product, and
Bus Engine OS is the rolling Linux distribution being built through it.

The superproject already had a modified `bus-engine-os` submodule pin before
this work resumed. I am leaving that line untouched and staging only the site,
docs, memo, and submodule-pin changes that come from this task.

## Work Narrative

I reread the repository guidance for the BusDK superproject, `busdk.com`, and
the public docs repository. The relevant constraints are that commercial pages
belong under `busdk.com/docs`, public operational documentation belongs under
the docs repo's `docs/` tree, and new `busdk.com` instructions must be recorded
in `busdk.com/PLAN.md` before editing. I also reread the product UI design
runbook because the product pages are user-facing surfaces; the rewrite should
stay clear, focused, and free of speculative controls or placeholder claims.

For the Codex App Server description I used the OpenAI docs skill and the
freshly cached Codex manual. The current manual describes Codex App Server as a
separate open-source Codex component using bidirectional JSON-RPC 2.0 with
stdio as the default transport and WebSocket/Unix socket options. That means the
Bus Engine copy should not call Codex App Server a REST service; if REST appears
in the architecture, it must be clearly described as Bus Engine's own API layer.

The implementation evidence points to available Engine CLI/API/event/artifact
and VM-provider surfaces, while `bus-engine-os` remains under active
development. Its README says Bus Engine currently uses Debian or Ubuntu as a
base and that Bus Engine OS is an incremental project toward a BusDK-controlled
Linux distribution. Its plan contains many implemented source/package recipes,
but still leaves rootfs, kernel, image, boot, update, and recovery work open.
Public copy therefore needs clear status labels instead of present-tense
promises about complete OS releases.

## Decisions

The new public copy will use "Bus Engine Founding Development Preview" and the
date-specific statement "Public development preview scheduled to begin on 1
July 2026" because today's date is 2026-06-23. I am also folding in the
operator's licensing constraint: the product must not promise public source-code
publication. Customer release areas may provide corresponding source to
recipients for GPL/LGPL/copyleft-covered binaries, while proprietary BusDK
userspace products remain binary-only where the applicable licenses permit it.
The website will keep this as a practical public summary and direct final
distribution-process details to legal review rather than overclaiming.

The operator added an important positioning correction while the rewrite was in
progress: the site must answer why Bus Engine is valuable compared with
installing an ordinary Linux distribution inside a normal virtual machine. The
answer is not VM management. Bus Engine's value is that it engineers and
maintains the operating system inside the VM around the workload, with the
system blueprint as the source of truth. I will add that section prominently to
the commercial overview and carry the same answer into the FAQ.

architecture page, development-images page, pricing page, FAQ, licensing page,

Linux VM. The pricing page has a static named-operator price summary so the page
remains useful if the Stripe pricing-table script is blocked.
