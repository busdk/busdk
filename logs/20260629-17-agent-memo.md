## Work Narrative

The immediate concern is commit order. Submodule contents must be committed in
their own repositories first, then the superproject can record the final
gitlinks and `.gitmodules` changes. I am inspecting each dirty module before
staging broadly so that the final handoff can say exactly which repositories
received commits and which checks, if any, were skipped.

I inspected the leaf repositories that the superproject reported as dirty. The
four new modules, `bus-api-provider-desktop`, `bus-integration-desktop`,
`bus-integration-systemd`, and `bus-portal-greeter`, were on `develop` with
only untracked bootstrap files. Each contained the same initial skeleton:
agent guidance, README/license/makefiles, `.gitignore`, and `.bus/dev`
workflow scripts. The `docs` and `sdd` modules each had four new empty module
documentation files matching those module names. I staged these six leaf
repositories first so their commits can be recorded by the superproject.

The leaf commits completed cleanly. The new module commits are `d87fcac` for
`bus-api-provider-desktop`, `40a9125` for `bus-integration-desktop`, `259395a`
for `bus-integration-systemd`, and `e76876e` for `bus-portal-greeter`. The

those commits, the superproject saw all six submodule pointers as updated, with

## Rebase Repair Follow-Up

Later in the same hour the superproject sync path failed again, this time on a
`bus-engine-os` rebase conflict. The submodule conflict was limited to
`PLAN.md`: remote had accepted more virtual GUI and terminal profile evidence,
while the local commit added unchecked planning tasks for renaming the public
GUI profile to `virtual-desktop` and for supporting named Engine instances. I
resolved the conflict by keeping the accepted remote evidence and placing the
new unchecked rename tasks at the start of the same planning section. The
submodule rebase produced `b90d638`, and the BusDK superproject rebase then
resolved its submodule gitlink conflict by recording that rebased commit.

After that rebase was repaired, rerunning `./scripts/sync-submodules.sh`
exposed a separate script bug. Three newly registered submodules were still
uninitialized, and the script launched their `git submodule update --init`
calls in parallel. Those concurrent commands all tried to update the BusDK
superproject git config and intermittently failed with a config lock message.
The fix is to initialize any missing or not-yet-materialized submodule targets
once, serially, before the parallel sync batch starts. The parallel fetch,
rebase, and push behavior remains unchanged after the submodule worktrees are
real.
