# 2026-06-30 21 UTC Agent Memo

The current goal shifted from the completed QEMU/WASM graphics and keyboard
MVP into the next Browser Lab product capability: Codex App Server running
inside the Bus Engine OS virtual server, reachable by frontend applications
and other WASM applications through a controlled browser-to-guest path.

I checked the relevant module guidance and plans before editing. QEMU's
`PLAN.md` had no open work because the previous graphics/keyboard MVP was
marked complete. That was correct for the old goal but wrong for the new one:
the next QEMU work is a generic service bridge, not more display work. I added
new active QEMU plan items for selecting a browser-to-guest channel, extending
the guest manifest, implementing a constrained browser API or iframe
`postMessage` surface, proving the path with a tiny generic guest service, and
documenting the downstream handoff. I left Bus Engine product behavior out of
QEMU.

In Bus Engine OS, I refined the active Browser Lab goal so the image must
include Codex App Server as a governed in-guest service, not only as a packaged
binary. The added plan items cover the service boundary, a systemd unit or
equivalent policy-gated service, a guest adapter that exposes only constrained
health/status/request shapes over the bridge, `virtual-server` and
`virtual-desktop` smoke proofs, and documentation for frontend and WASM apps.
The package recipes for Codex CLI and Codex App Server already exist for
aarch64 and x86_64, but their license status remains pending until bundled
helper notices are reviewed.

I also updated the Bus Engine CLI, API provider, Engine integration, VM
integration, QEMU provider, and VM API provider plans so the system has a
coherent control surface. The product-facing layers should report stable
bridge and service-health states, preserve default single-instance behavior,
support named instances later, and avoid exposing QEMU transport details,
guest filesystem paths, raw port forwarding, arbitrary command execution, or
model credentials.

Verification so far is planning-level only. `git diff --check` passed in QEMU,
Bus Engine OS, `bus-engine`, `bus-api-provider-engine`,
`bus-integration-engine`, `bus-integration-vm`, `bus-integration-qemu`, and
`bus-api-provider-vm`. No implementation or runtime proof was attempted in this
phase.

Next: commit the plan updates in each repository, then record the updated
submodule pins in the BusDK superproject. The active goal is now the in-guest
Codex App Server Browser Lab bridge.
