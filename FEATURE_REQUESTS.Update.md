# FEATURE_REQUESTS.md

Enhancement requests for BusDK in this repo context.

Privacy rule for request write-ups:
- Keep examples/repro snippets sanitized (no real customer names/emails/IBANs/account numbers/invoice numbers/local paths).
- Prefer placeholders and aggregated outputs over raw customer-linked row dumps.

Last reviewed: 2026-03-10.

Goal note:
- Target workflow is Bus-only for bookkeeping/audit operations.
- Operators should not need shell text pipelines (`grep/sed/awk/column`) to answer accounting control questions.

Active requests:

1. Replace the fragmented Finnish reporting configuration model with a layered, deterministic reporting architecture aligned with good accounting practice.
   - Design basis:
     - the target model is described in `original/tutkimustietoa/raportoinnin-tilien-konfigurointi.md`
     - the key principle is to separate:
       - statutory reporting taxonomy
       - account semantics / classification
       - workspace entity context
       - company-specific overrides
   - Current behavior:
     - workspace-level reporting configuration in `bus config` only covers high-level profile choices such as:
       - `reporting_standard`
       - `income_statement_scheme`
       - `comparatives`
       - `prepared_under_pma`
     - `bus-accounts` currently owns `report-account-mapping.csv`, where the effective reporting meaning of an account is expressed against layout-specific keys such as:
       - `layout_id`
       - `account_code`
       - `statement_target`
       - `layout_line_id`
       - `normal_side`
       - `rollup_rule`
     - `bus-reports` also contains built-in layout defaults and prefix/type-based fallback logic, plus a separate `report-layout-label-overrides.csv` mechanism for wording overrides.
     - `fi-*` reporting still effectively depends on layout-specific account mapping materialization, validation, or both, so the operator ends up curating per-layout presentation logic instead of only company-specific bookkeeping meaning.
     - this is not just awkward but conceptually wrong for Finnish bookkeeping:
       - a layout-specific mapping file is not a lawful or sound substitute for a real chart-of-accounts hierarchy and canonical account-group model
       - the bookkeeping meaning of an account should not live primarily in a report-layout override table
     - `accounts.csv` may contain extra reporting/control fields, but today those are still maintained by direct CSV editing rather than a first-class reporting-classification model.
     - the same business meaning can therefore be duplicated across:
       - workspace profile in `datapackage.json`
       - `accounts.csv`
       - `report-account-mapping.csv`
       - optional label-override CSV
       - built-in report layout logic
     - this is difficult to reason about and audit because one account may need to be restated separately for:
       - KPA vs PMA variants
       - normal statement vs `*-accounts` drill-down variants
       - custom layouts vs built-in layouts
   - Requested behavior:
     - redesign Finnish reporting around explicit layers:
       - it is acceptable to do this as a large refactor instead of preserving the current layout-centric implementation shape for backward compatibility
       - compatibility with today's weak account-under-layout/account-under-account patterns is secondary to ending up with a correct Finnish bookkeeping model
       - a versioned statutory taxonomy package:
          - KPA
          - PMA
          - and supported special schemes such as real-estate / housing-company / association-foundation variants
       - a canonical account-classification layer independent of output layout:
         - semantic tags or categories such as cash, receivables, VAT payable, personnel expenses, depreciation, financial income
       - a first-class hierarchical account-group model in `bus-accounts`:
         - account groups must be separate objects, not accounts pretending to be parent rows for other accounts
         - each group must have:
           - a stable identifier
           - a presentation code / number
           - a name
           - an optional parent group
         - top-level groups may be ungrouped roots with no parent
         - every posting account must belong to exactly one account group
         - account groups must be the normal canonical structure used to express Finnish chart hierarchy for:
           - TASE
           - tuloslaskelma
           - tililuettelo and account-balance style outputs
           - VAT-processing defaults
           - other account-family-level reporting and control rules
         - group numbering must not be modeled with the same uniqueness rules as account numbering:
           - posting account codes remain globally unique
           - group codes are presentation codes and may repeat on an ancestor/descendant path
           - deterministic identity must therefore be something like stable `group_id` or full parent-scoped path, not just raw group code
           - sibling collisions under the same parent must still be invalid
           - for example:
             - allowed: `200 -> 200`, `200 -> 210`
             - invalid: two sibling branches both `200 -> 210`
         - Bus should support trees like:
           - `100 Vastaavaa`
           - `100 > 100 Pysyvät vastaavat`
           - `100 > 150 Vaihtuvat vastaavat`
           - `100 > 150 > 150 Vaihto-omaisuus`
           - `100 > 150 > 150 > 1501 Aineet ja tarvikkeet`
         - group-level metadata must support statement and control behavior such as:
           - visible in full income statement
           - visible in short/compact income statement
           - VAT-processing defaults for accounts under that group
           - other semantic/reporting flags where inheritance from group to account is the natural default
         - top summary groups do not need the same visibility flags as leaf/detail groups, so the model must distinguish structural grouping from display behavior
       - a workspace entity-context layer:
         - legal form
         - size class
         - reporting framework
         - industry/special flags
         - language / presentation unit
         - other filing-relevant context
       - an explicit company-override layer for exceptional accounts or presentation choices only
     - use deterministic resolution precedence:
       - account-specific override
       - account-group rule / inherited group metadata
       - semantic classification rule
       - numbering-family or range default
       - hard validation error when still unresolved
     - make the built-in Finnish defaults good enough that a normal small-company workspace can usually produce correct statutory reports after setting only entity context and verifying a small number of exceptions.
     - ensure `*-accounts` and other drill-down views inherit the same effective classification as the base statement; detail level must not redefine statement meaning.
     - remove `report-account-mapping.csv` entirely from the normal Finnish reporting architecture:
       - the everyday chart hierarchy must live in first-class account-group data owned by `bus-accounts`
       - statutory reporting should resolve from taxonomy + entity context + account-group/account semantics + explicit exceptional overrides, without a layout-keyed per-account mapping CSV
       - Bus should not encourage modeling Finnish tiliryhmät by attaching one account under another account
       - if compatibility with old workspaces is needed, provide a one-time migration/import path instead of preserving `report-account-mapping.csv` as a first-class long-term contract
     - treat wording/label customization as taxonomy/profile customization, not as a separate ad hoc reporting side file unless the workspace explicitly needs a rare override.
     - keep the model versioned and migratable so future KPA/PMA/iXBRL/XBRL changes can be adopted explicitly instead of leaking into company-local mapping files.
     - if keeping old datasets would materially compromise the new model, Bus should prefer a clean replacement plus an explicit migration/import tool over preserving a misleading contract indefinitely.
   - Operational goals:
     - reports should be easy to generate in correct Finnish form with as little company-specific configuration as possible.
     - the normal operator workflow should be closer to:
       - initialize or import the account-group tree
       - set entity context
       - review suggested account-group membership and account classification
       - add explicit overrides only for exceptions
       - generate reports
     - Bus should provide first-class account-group tooling, not only raw CSV editing:
       - create/init/list/tree/set/validate/explain operations for groups and account-to-group membership
       - read/query commands must also be group-aware, not only configuration commands
       - account listing and balance aggregation must support querying by account group
       - account listing CLI must also support opening and closing balances for each listed account
       - balance-oriented outputs should be able to show at least:
         - opening balance
         - period movement or net change
         - closing balance
       - for human-facing account-list/tree output, the default row layout should be deterministic:
         - latest / closing balance is shown immediately after the account name
         - opening balance is shown as the rightmost final balance column
         - when intermediate period checkpoints are requested, their opening balances may appear between those two
       - in practice this means an account row can read conceptually as:
         - `1910 Pankkitili 1234,56 ... 1000,00`
         - where `1234,56` is the latest/closing balance next to the name and `1000,00` is the opening balance at the far right
       - the same ordering principle should apply consistently to group subtotal rows, not only leaf account rows
       - the operator must be able to ask for one group and get:
         - the selected group
         - descendant subgroups
         - the accounts belonging to each shown subgroup
         - subtotal for each subgroup/path
         - total for the full selected subtree
         - opening and closing balances at both account level and shown group/path subtotal level
       - this should work for both structure-only listing and journal-derived balance sums
       - example target workflows:
         - list all accounts under group `200` including child groups and accounts
         - calculate balances for group `200` including `200 > 210 > ...` descendants
         - show the tree with path totals so each displayed branch total is visible without external spreadsheet work
       - Bus should provide a native tree-oriented output mode for this, rather than forcing operators to post-process flat TSV/CSV manually
       - CLI must support deterministic bulk membership assignment, not only one account at a time
       - acceptable bulk selectors should include at least:
         - exact account code
         - account-code prefix families such as `2*` or `2xxx`
         - explicit numeric ranges such as `2000-2999`
       - example target workflow:
         - assign all `2xxx` accounts to account group `200`
         - then refine narrower subfamilies such as `29xx` into a child group when needed
       - precedence must stay deterministic:
         - exact account assignment overrides broader family/range assignment
         - narrower range/prefix overrides broader range/prefix
         - equal-priority ambiguity is a validation error
       - deterministic migration helpers from current flat `accounts.csv` plus legacy reporting datasets such as `report-account-classification.csv` / `report-account-mapping.csv`
     - Bus should provide deterministic explain/validate tooling that answers:
       - why each account landed on a given statement line
       - which account group it belongs to and through which group path it inherited reporting or VAT behavior
       - which rule or override decided it
       - which accounts remain unresolved
     - the architecture should cover the full close/reporting surface coherently:
       - balance sheet
       - profit and loss
       - tililuettelo / account balances
       - tase-erittely / balance-sheet specification
       - balance-sheet reconciliation
       - evidence-pack / filing-facing artifact generation
   - Why this matters:
     - good accounting practice requires that the statutory presentation model, the bookkeeping meaning of accounts, and company-specific exceptions remain understandable and auditable.
     - the current model is too layout-centric and duplicates meaning across multiple files and built-in logic layers.
     - operators should not have to rebuild Bus's reporting semantics by hand for each workspace just to get legally correct statements.
     - a layered model would make the old layout-keyed mapping-file approach obsolete rather than worth extending further.

       - `tase-accounts-<as-of>.pdf`
       - `tase-erittelyt-<as-of>.pdf`
       - `tase-tasmaytys-<as-of>.pdf`
       - `tositeluettelo-<period>.csv` and `.pdf`
       - `pankkitapahtumat-<period>.csv` and `.pdf`
       - `day-book-<period>.pdf`
       - `general-ledger-<period>.pdf`

     - add a way to declare the exact artifact profile for the final package:
       - include/exclude specific artifact families
       - include/exclude specific variants
       - disable Markdown outputs entirely when not wanted
       - optionally choose deterministic filename templates / naming style
     - ideally support either:
       - a named built-in artifact profile for the Finnish accountant/year-end package, or
       - explicit include/exclude/profile/naming flags for evidence-pack artifact variants and layouts
   - Why this matters:
     - Bus now already owns most of the package boundary, so the remaining value is to let one native command produce exactly the approved final file set.
     - repo-local scripts should not need to hand-delete unwanted Markdown or hand-rename outputs after evidence-pack.
     - the accountant-facing package should come from one deterministic native command and already match the target artifact profile.

3. Add workspace-level configuration for how Bus generates all bookkeeping-facing IDs.
   - Current behavior:
     - Bus currently generates opaque technical IDs such as:
       - `2026-03-07T144331Z-hzzox2`
     - these appear in outputs like `voucher-list` as:
       - `document_number`
       - `voucher_id`
       - `transaction_id`
     - in narrow report columns and PDFs, these long timestamp-based IDs are hard to read and can visually truncate so that many rows appear to start the same way.
   - Observed accounting-style expectation:
     - the historical/manual bookkeeping model used short human-readable voucher numbering, for example:
       - `1`
       - `2`
       - `9`
       - `H-1`
       - `H-2`
     - those identifiers acted as meaningful voucher/document numbers in the day-book instead of opaque runtime-generated tokens.
   - Requested behavior:
     - allow the workspace to define a central ID generation policy for every bookkeeping-facing ID type that Bus emits or derives.
     - this should cover at least:
       - `document_number`
       - `voucher_id`
       - `transaction_id`
       - any other report-visible bookkeeping IDs created by Bus for vouchers, report rows, transactions, journals, or related posting groups
     - the policy should allow, for each ID type as needed:
       - selectable generation strategy
       - configurable format/template expression
       - numbering scheme
       - prefixes/suffixes
       - one or more explicit number ranges / reserved ranges per series
       - journal/source/channel specific series
       - yearly or period-local sequences
       - human-readable format separate from any internal opaque key if Bus still needs one
     - make the policy deterministic and replay-safe.
     - the default strategy should be “next highest after existing highest value in the configured series/range”.
     - alternative strategies should be configurable, for example:
       - incrementing number
       - timestamp/date-based parts
       - combined formats built from multiple parts
     - the format should be definable as a formula/template, for example:
       - `M-{inc}`
       - `M-{yyyy}-{inc}`
       - `TX-{yyyyMMdd}-{inc}`
       - or other built-in deterministic placeholders
   - Example configuration goals:
      - plain yearly running numbers: `1`, `2`, `3`, ...
      - prefixed series such as `H-1`, `H-2`, `P-1`
      - range-constrained series such as:
        - purchase invoices `P10000` ... `P99999`
        - sales invoices `M1000` ... `M9999`
      - multiple configured ranges for the same logical series, for example:
        - sales invoices `M1000 ... M9999`
        - and later `M100000 ... M999999`
      - in normal use there may be only one configured range, but the workspace should be allowed to define multiple future/overflow ranges in advance
      - default selection rule:
        - inspect existing IDs in the relevant configured series
        - choose the next largest allowed value
      - custom composed format examples:
        - `M-{yyyy}-{inc}`
        - `P-{yyyy}-{inc}`
        - `V-{period}-{inc}`
      - short readable TX identifiers instead of long timestamp-random tokens
      - separate human-readable visible IDs while retaining an internal stable UUID-like key if Bus still needs one
   - Why this matters:
      - bookkeeping reports are accountant-facing documents, not just internal runtime traces
      - human-readable IDs are materially more usable in PDFs, review, audit, and cross-reference against legacy/manual books
      - the numbering policy should be a workspace configuration concern, not an implicit built-in generator with one hardcoded format
      - in real bookkeeping workflows, different document classes often need separate reserved numbering spaces, not just different prefixes
      - a single numbering series may also need multiple non-contiguous allowed ranges over time, so the configuration model should support more than one range per series
